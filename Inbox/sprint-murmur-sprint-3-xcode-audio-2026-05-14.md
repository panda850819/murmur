## Stuck point (synthesized from PAUSED1 + restructure attempt)

`xcodebuild` cannot make WhisperKit's swiftmodule reachable to source that
imports it, regardless of project structure tried.

The Xcode build does compile WhisperKit (we see SwiftDriver output for it) and
resolves the package graph cleanly (`xcodebuild -resolvePackageDependencies`
succeeds and lists argmax-oss-swift @ 1.0.0). But `import WhisperKit` fails
to find the module at MurmurCore compile time.

## Path A is confirmed-blocked

The XcodeGen 2.45.4 issue compounds:

1. Initial bug: local package product dep has no `package = <ref>` link in pbxproj. **Workaround:** post-gen sed/Python patch (`scripts/patch-xcodeproj.py` exists and works for this layer).
2. Deeper bug: even with linkage patched and WhisperKit explicitly declared at the Xcode target level, the LOCAL SPM target's compile context doesn't get WhisperKit's module search path / framework linkage. This is the Xcode↔local-SPM↔remote-SPM transitive linkage layer, which isn't fixable from project.yml or a simple pbxproj sed.

Possible fixes from here, none small:

- **Tuist** instead of XcodeGen (different project generator, may handle this case correctly)
- Hand-edit pbxproj more invasively to add WhisperKit to MurmurCore's compile sources / framework search paths (very brittle)
- Drop XcodeGen entirely, hand-craft `Murmur.xcodeproj`, add WhisperKit via Xcode UI (= Path B)
- Drop the local-SPM-package strategy entirely; have Xcode project compile MurmurCore sources directly + add WhisperKit at Xcode level (= Path B variant)
- Compile MurmurCore sources into MurmurMac target (single Xcode target), add WhisperKit at Xcode target level, KEEP Core/Package.swift only for `swift test` and any future iOS share (hybrid)
